Тестирование бизнес-процесса ЭТрН

# Чек-лист тестирования бизнес-процесса ЭТрН

> **Формат файла:** КПГ (Квитанция о приёме груза для перевозки), XSD-версия 5.01, КНД 1110442  
> **Охват:** BPMN-процесс (4 титула), XSD-валидация, ролевая модель, негативные и граничные сценарии

---

## 1. Проверка BPMN-схемы и бизнес-процесса

### 1.1 Общая целостность процесса

- [ ] Процесс имеет единственную точку старта.
- [ ] Все ветки процесса завершаются конечным событием.
- [ ] Отсутствуют «висящие» переходы без продолжения.
- [ ] Для каждого шлюза (Gateway) определены все возможные исходы.
- [ ] Отсутствуют циклические переходы без условий выхода.
- [ ] Все события получения сообщений имеют соответствующие события отправки.
- [ ] Все переходы между участниками процесса логически обоснованы.
- [ ] Последовательность титулов (1 → 2 → 3 → 4) соответствует бизнес-логике документооборота.
- [ ] Каждый титул формируется строго после завершения предыдущего этапа процесса.
- [ ] Прерывающее событие «Отказ от груза» (ветка «Нет» при приёмке на разгрузку) корректно завершает процесс.

### 1.2 Проверка ролей участников

- [ ] Грузоотправитель может создать ЭТрН.
- [ ] Перевозчик получает документ (Титул 1) после отправки грузоотправителем.
- [ ] Перевозчик может подтвердить получение документа.
- [ ] Диспетчер получает данные о погрузке от водителя.
- [ ] Грузополучатель получает Титул 2 и Титул 3.
- [ ] Водитель получает QR-код после формирования Титула 2.
- [ ] ГИС ЭПД получает Титул 3 (отдельный канал доставки).
- [ ] Каждый участник имеет доступ только к своим операциям в соответствии с ролевой моделью.
- [ ] Грузоотправитель не может выполнить операции перевозчика и наоборот.

---

## 2. Создание ЭТрН (Титул 1)

### 2.1 Позитивные сценарии

- [ ] Создание ЭТрН с заполнением всех обязательных реквизитов.
- [ ] Создание ЭТрН с заполнением обязательных и необязательных реквизитов.
- [ ] Сохранение версии документа как черновик.
- [ ] Повторное открытие сохранённого черновика.
- [ ] Редактирование сохранённого черновика.
- [ ] Подписание документа ЭП и отправка Титула 1.
- [ ] После отправки Титула 1 устанавливается статус ЭТрН «Ожидает отгрузки со склада».
- [ ] Титул 1 корректно маршрутизируется: достигает перевозчика через «Маршрутизацию сообщений».

### 2.2 Негативные сценарии

- [ ] Попытка создания ЭТрН без обязательных реквизитов — система выдаёт ошибку валидации.
- [ ] Попытка подписания документа с незаполненными обязательными полями — отказ с описанием ошибок.
- [ ] Попытка отправки неподписанного Титула 1 — отказ.
- [ ] Попытка создать Титул 2 до отправки Титула 1 — недоступно.
- [ ] Попытка грузоотправителя повторно отправить уже отправленный Титул 1.

---

## 3. Маршрутизация сообщений

- [ ] После получения Титула 1 сервис «Маршрутизация сообщений» корректно определяет получателей.
- [ ] Сообщение доставляется перевозчику.
- [ ] При недоступности получателя система сохраняет сообщение и повторяет доставку.
- [ ] Дублирование сообщения при повторной отправке не приводит к созданию двух документов у получателя.

---

## 4. Подтверждение получения перевозчиком

### 4.1 Позитивные сценарии

- [ ] Перевозчик может подтвердить получение Титула 1.
- [ ] После подтверждения статус документа корректно изменяется.
- [ ] Действие фиксируется в истории документа с датой, временем и пользователем.

### 4.2 Негативные сценарии

- [ ] Повторное подтверждение получения невозможно.
- [ ] Пользователь без роли «Перевозчик» не может подтвердить получение.
- [ ] Подтверждение невозможно, если Титул 1 ещё не получен системой.

---

## 5. Получение информации о погрузке (формирование Титула 2)

### 5.1 Без замечаний при погрузке

- [ ] Водитель прибывает на погрузку — событие «Водитель прибыл на погрузку» инициирует процесс.
- [ ] Контроль погрузки выполнен — данные о погрузке передаются диспетчеру.
- [ ] Формируется и подписывается Титул 2 без замечаний.
- [ ] Титул 2 отправляется грузоотправителю и грузополучателю.
- [ ] Генерируется QR-код и отправляется водителю.

### 5.2 С замечаниями при погрузке

- [ ] Доступен чекбокс «Замечания при погрузке».
- [ ] При установке чекбокса становятся доступны поля описания замечаний (несоответствий).
- [ ] Замечания сохраняются корректно.
- [ ] Замечания отображаются в Титуле 2.
- [ ] Формируется и подписывается Титул 2 с информацией о замечаниях.
- [ ] Титул 2 с замечаниями отправляется грузоотправителю и грузополучателю.

### 5.3 Негативные сценарии

- [ ] Установлен чекбокс замечаний, но поля описания не заполнены — система блокирует отправку.
- [ ] Превышена максимальная длина описания замечаний — выдаётся ошибка.
- [ ] Попытка сохранить недопустимые символы в описании замечаний.
- [ ] Попытка сформировать Титул 2 без прохождения этапа контроля погрузки.

---

## 6. Второй титул

- [ ] Формируется только после обработки информации о погрузке.
- [ ] Содержит корректные сведения о погрузке (с замечаниями или без).
- [ ] Отправляется грузоотправителю.
- [ ] Отправляется грузополучателю.
- [ ] QR-код генерируется и отправляется водителю одновременно с рассылкой Титула 2.
- [ ] После получения Титула 2 устанавливается статус ЭТрН «Груз принят к перевозке».
- [ ] Титул 2 отображается в истории документа.

---

## 7. Приёмка на разгрузку (ожидание ТС на месте разгрузки → Титул 3)

### 7.1 Позитивные сценарии — «Принять груз: Да»

- [ ] Водитель прибывает на место разгрузки — фиксируется событие «Сообщить о прибытии на разгрузку».
- [ ] Грузополучатель ожидает ТС на месте разгрузки.
- [ ] При решении «Принять груз: Да» активируется чекбокс «Груз принят».
- [ ] Заполняются данные о приёмке.
- [ ] Подписывается и отправляется Титул 3.
- [ ] Титул 3 отправляется грузоотправителю и перевозчику.
- [ ] Титул 3 отправляется в ГИС ЭПД (отдельный канал).
- [ ] После получения Титула 3 устанавливается статус ЭТрН «Груз принят грузополучателем».

### 7.2 Негативные сценарии — «Принять груз: Нет» (Отказ от груза)

- [ ] При решении «Нет» процесс завершается по ветке «Отказ от груза».
- [ ] Статус документа корректно отражает отказ.
- [ ] Попытка сформировать Титул 3 после отказа — недоступно.
- [ ] Грузополучатель не может изменить решение «Нет» на «Да» без отдельного процесса.

---

## 8. Третий титул

- [ ] Формируется только после подтверждения приёмки грузополучателем.
- [ ] Содержит корректные сведения о приёмке.
- [ ] Отправляется грузоотправителю и перевозчику.
- [ ] Отправляется в ГИС ЭПД.
- [ ] Корректно отображается всем участникам процесса согласно ролям.
- [ ] После отправки Титула 3 становится доступен этап разгрузки/получения сведений о разгрузке.

---

## 9. Разгрузка и формирование Титула 4

### 9.1 Получение сведений о разгрузке

- [ ] Фиксируется ожидание разгрузки («Дождаться разгрузки»).
- [ ] После разгрузки диспетчер получает итоги разгрузки («Доложить об итогах разгрузки диспетчеру»).
- [ ] Сведения о разгрузке получены («Получить сведения о разгрузке»).

### 9.2 Без замечаний при выгрузке

- [ ] При отсутствии замечаний (Gateway «Нет») подписывается и отправляется Титул 4 без полей несоответствий.

### 9.3 С замечаниями при выгрузке

- [ ] Активируется чекбокс «Замечания при выгрузке».
- [ ] Становятся доступны поля несоответствий — данные вводятся корректно.
- [ ] Подписывается и отправляется Титул 4 с замечаниями при выгрузке.

### 9.4 Негативные сценарии

- [ ] Установлен чекбокс замечаний при выгрузке, но поля не заполнены — блокировка отправки.
- [ ] Попытка сформировать Титул 4 без этапа разгрузки — недоступно.
- [ ] Попытка сформировать Титул 4 без Титула 3 — недоступно.

---

## 10. Четвёртый титул

- [ ] Формируется только после завершения разгрузки и получения сведений.
- [ ] Содержит итоговое состояние перевозки (с замечаниями или без).
- [ ] После получения Титула 4 устанавливается финальный статус ЭТрН «Завершён».
- [ ] Титул 4 рассылается грузоотправителю (ГО) и грузополучателю (ГП).
- [ ] Документ становится недоступным для редактирования после завершения.
- [ ] Завершённый документ доступен для просмотра всем участникам процесса.

---

## 11. Проверка истории статусов

- [ ] Все изменения статусов сохраняются в хронологическом порядке.
- [ ] Статусы отображаются в правильной последовательности.
- [ ] Для каждого статуса фиксируются дата и время изменения.
- [ ] Для каждого статуса фиксируется инициатор изменения (пользователь/роль).
- [ ] Присутствуют все промежуточные статусы: «Ожидает отгрузки со склада» → «Груз принят к перевозке» → «Груз принят грузополучателем» → «Завершён».
- [ ] Статус недоступен для ручного изменения пользователем в обход бизнес-процесса.

---

## 12. QR-код для водителя

- [ ] QR-код генерируется корректно после отправки Титула 2.
- [ ] QR-код доставляется водителю.
- [ ] QR-код содержит актуальные данные документа.
- [ ] Водитель успешно получает QR-код («Получить QR-код»).
- [ ] QR-код не генерируется повторно при дублирующей отправке Титула 2.
- [ ] QR-код корректно считывается при прибытии на место разгрузки.

---

## 13. Интеграция с ГИС ЭПД

- [ ] Титул 3 отправляется в ГИС ЭПД отдельным каналом (не совмещается с рассылкой участникам).
- [ ] Успешная доставка в ГИС ЭПД подтверждается.
- [ ] При недоступности ГИС ЭПД система сохраняет попытку и выполняет повторную отправку.
- [ ] Ошибка интеграции с ГИС ЭПД не блокирует основной бизнес-процесс участников.

## 14. Проверки XSD-схемы (КНД 1110442, версия 5.01)

### 14.1 Элемент `ИдПолИной` (идентификатор иного получателя)

- [ ] Минимум 4 символа — значение в 3 символа отклоняется.
- [ ] Максимум 46 символов — значение в 47 символов отклоняется.
- [ ] Значение ровно 4 символа — граничное значение — принимается.
- [ ] Значение ровно 46 символов — граничное значение — принимается.
- [ ] Элемент необязателен (minOccurs=0) — отсутствие принимается.
- [ ] Несколько элементов `ИдПолИной` — принимается (maxOccurs=unbounded).

### 14.2 Атрибуты `Документ`

- [ ] `УИД_КПГ`: минимум 1, максимум 36 символов.
- [ ] `НомКПГ`: минимум 1, максимум 50 символов.
- [ ] `ДатаКПГ`: корректный формат ДД.ММ.ГГГГ.
- [ ] `ПортОтпр`: минимум 1, максимум 500 символов.
- [ ] `ПортНазн`: минимум 1, максимум 500 символов.
- [ ] `ДатГрузПрвз` (дата приёма груза к перевозке): обязателен, формат ДД.ММ.ГГГГ.
- [ ] `ДатГрузХран` (дата приёма груза на хранение): необязателен — отсутствие принимается.
- [ ] `НаимДокИнфПрв`: минимум 1, максимум 255 символов.
- [ ] `ДатИнфПрв`: обязателен, формат ДД.ММ.ГГГГ.
- [ ] `ВрИнфПрв`: обязателен, формат ЧЧ:ММ:СС.

### 14.3 Элемент `СвГруз` → атрибуты `ОпГруз`

- [ ] `НаимГруз` (обязателен): минимум 1, максимум 255 символов.
- [ ] `ВесГрузБрут` (обязателен): decimal, totalDigits=18, fractionDigits=4 — корректное значение принимается.
- [ ] `ВесГрузБрут`: значение с 5 знаками после запятой — отклоняется.
- [ ] `ВесГрузБрут`: отрицательное значение — проверить поведение (XSD не ограничивает знак).
- [ ] `ВесГрузБрут`: нулевое значение `0` — проверить бизнес-логику.
- [ ] `ВесГрузБрут`: буквенное значение — отклоняется.
- [ ] `ПорНомГруз`: необязателен, totalDigits=5 — превышение отклоняется.
- [ ] `НомМаркГруз`: необязателен, минимум 1, максимум 30 символов.
- [ ] `КодТовНом`: необязателен, минимум 4, максимум 10 символов.
- [ ] `Объем`: decimal, totalDigits=18, fractionDigits=6 — корректное значение принимается.
- [ ] `Объем`: 7 знаков после запятой — отклоняется.
- [ ] `КолМестГруз`: необязателен, totalDigits=11.
- [ ] `ВесБрутКонтейн`: необязателен, fractionDigits=4.
- [ ] `КолПаллет`: необязателен, totalDigits=8.
- [ ] `ВидТар` (ОКВГУМТип): ровно 2 символа — иное отклоняется.

### 14.4 Элемент `ОпГруз` → `КодКласОпасн` (опасные грузы)

- [ ] Минимум 1 символ, максимум 3 символа.
- [ ] Элемент необязателен (minOccurs=0) — отсутствие принимается.
- [ ] Несколько элементов — принимается (maxOccurs=unbounded).


### 14.5 Элемент `Габар` (габариты груза в метрах)

- [ ] `ВысЗнач`, `ДлЗнач`, `ШирЗнач`: все обязательны.
- [ ] Decimal: totalDigits=5, fractionDigits=3 — значение `99.999` принимается.
- [ ] Значение с 4 знаками после запятой — отклоняется.
- [ ] totalDigits=5: значение `100.00` (6 значимых цифр) — отклоняется.
- [ ] Элемент `Габар` необязателен — отсутствие принимается.


### 14.6 Объявленная стоимость груза `ОбЦеннГр` / `ЦеннГрузТип`

- [ ] Должен присутствовать ровно один из элементов: `СтЦеннГрНалог` или `СтЦеннГр` (xs:choice).
- [ ] Наличие обоих элементов одновременно — отклоняется.
- [ ] Отсутствие обоих элементов — отклоняется.
- [ ] `КодОКВ` (обязателен): ровно 3 цифры (ОКВТип).
- [ ] `НаимОКВ` (обязателен): минимум 1, максимум 100 символов.
- [ ] Decimal для стоимости: totalDigits=19, fractionDigits=2.

### 14.7 Элемент `СвСопрДок` (сопроводительные документы)

- [ ] Минимум один элемент обязателен (maxOccurs=unbounded).
- [ ] `РеквНаимДок`, `РеквНомерДок`, `РеквДатаДок` — все обязательны.
- [ ] `РеквНаимДок`: минимум 1, максимум 255 символов.
- [ ] `РеквНомерДок`: минимум 1, максимум 255 символов.
- [ ] `РеквДатаДок`: формат ДД.ММ.ГГГГ.
- [ ] `РеквДопСведДок` (необязателен): максимум 2000 символов — наибольший лимит из всех строковых полей.
- [ ] Несколько элементов `СвСопрДок` — принимается.

### 14.8 Подписант `ПодпИнфПрв` / `ПодписантТип`

- [ ] `Должн` (обязателен): минимум 1, максимум 255 символов.
- [ ] `СпосПодтПолном` (обязателен): допустимые значения `1`, `2`, `3`, `4`, `5`, `6` — иные отклоняются.
- [ ] `ТипПодпис` (необязателен): допустимые значения `1`, `2`, `3` — значение `4` отклоняется.
- [ ] При `СпосПодтПолном = 3`: элемент `СвДоверЭл` **обязателен** (Schematron: код 0400300003).
- [ ] При `СпосПодтПолном = 3`: элемент `СвДоверБум` **должен отсутствовать**.
- [ ] При `СпосПодтПолном = 5`: элемент `СвДоверБум` **обязателен** (Schematron: код 0400300003).
- [ ] При `СпосПодтПолном = 5`: элемент `СвДоверЭл` **должен отсутствовать**.
- [ ] При `СпосПодтПолном` ≠ 3 и ≠ 5: оба элемента `СвДоверЭл` и `СвДоверБум` **должны отсутствовать**.
- [ ] `ДатаПодДок` (необязателен): формат ДД.ММ.ГГГГ.
- [ ] `ДопСведПодп` (необязателен): максимум 255 символов.

## 15. Проверки типовых полей (Boundary Value Analysis)

### 15.1 ИНН

- [ ] ИНН ЮЛ из 10 цифр, соответствующий паттерну `([0-9]{1}[1-9]{1}|[1-9]{1}[0-9]{1})[0-9]{8}`, — принимается.
- [ ] ИНН ФЛ из 12 цифр, соответствующий паттерну `([0-9]{1}[1-9]{1}|[1-9]{1}[0-9]{1})[0-9]{10}`, — принимается.
- [ ] ИНН из 9 символов — отклоняется.
- [ ] ИНН из 11 символов — отклоняется.
- [ ] ИНН, начинающийся с `00`, — отклоняется (паттерн требует ненулевой первой или второй цифры).
- [ ] ИНН с буквами — отклоняется.
- [ ] ИНН со спецсимволами — отклоняется.

### 15.2 КПП

- [ ] КПП из 9 символов, соответствующий паттерну, — принимается.
- [ ] КПП из 8 символов — отклоняется.
- [ ] КПП из 10 символов — отклоняется.
- [ ] КПП с допустимыми буквами (A-Z) в 5-6 позиции — принимается.
- [ ] КПП с кириллицей в 5-6 позиции — отклоняется.

### 15.3 Дата (ДатаТип — ДД.ММ.ГГГГ)

- [ ] Корректная дата `01.01.2024` — принимается.
- [ ] `29.02.2024` (2024 — високосный год) — принимается.
- [ ] `29.02.2023` (2023 — невисокосный год) — **отклоняется**.
- [ ] `31.01.2024` — принимается.
- [ ] `31.04.2024` — **отклоняется** (апрель — 30 дней).
- [ ] `00.01.2024` — отклоняется.
- [ ] `32.01.2024` — отклоняется.
- [ ] `01.13.2024` — отклоняется.
- [ ] `01.00.2024` — отклоняется.
- [ ] `01.01.1899` — отклоняется (диапазон 1900–2099).
- [ ] `01.01.2100` — отклоняется (диапазон 1900–2099).
- [ ] Формат `2024-01-01` — отклоняется.
- [ ] Формат `01/01/2024` — отклоняется.

### 15.4 Время (ВремяТип — ЧЧ:ММ:СС)

- [ ] `00:00:00` — принимается.
- [ ] `23:59:59` — принимается.
- [ ] `24:00:00` — **отклоняется**.
- [ ] `12:60:00` — **отклоняется**.
- [ ] `12:00:60` — **отклоняется**.
- [ ] `1:00:00` (без ведущего нуля) — отклоняется (length=8).
- [ ] Пустая строка — отклоняется.

### 15.5 БИК

- [ ] Ровно 9 цифр, паттерн `[0-9]{9}` — принимается.
- [ ] 8 цифр — отклоняется.
- [ ] 10 цифр — отклоняется.
- [ ] Содержит букву — отклоняется.

### 15.6 Почтовый индекс (в `АдрРФТип`)

- [ ] Ровно 6 цифр, паттерн `[0-9]{6}` — принимается.
- [ ] 5 цифр — отклоняется.
- [ ] 7 цифр — отклоняется.
- [ ] Содержит букву — отклоняется.

### 15.7 ОГРНИП

- [ ] Ровно 15 цифр, паттерн `[0-9]{15}` — принимается.
- [ ] 14 цифр — отклоняется.
- [ ] 16 цифр — отклоняется.

### 15.8 Адрес ГАР (`АдрГАРТип`)

- [ ] `ИдНом` (GUID): ровно 36 символов — принимается.
- [ ] `ИдНом` из 35 или 37 символов — отклоняется.
- [ ] `ГЛН` (GLN): ровно 13 символов — 12 и 14 отклоняются.
- [ ] При `КодРегион` ≠ `99` элемент `МуниципРайон` **обязателен** (Schematron).
- [ ] `КодРегион`: ровно 2 цифры (ССРФТип) — 1 и 3 цифры отклоняются.

### 15.9 Участники (`УчастникТип`) — взаимоисключающий выбор `xs:choice`

- [ ] Допускается ровно один из вариантов: `СвИП`, `СвЮЛУч`, `СвИнНеУч`, `СвФЛУч`.
- [ ] Два варианта одновременно — отклоняется.
- [ ] Отсутствие всех вариантов — отклоняется.

### 15.10 `СвФЛУч` — условная обязательность (Schematron)

- [ ] При отсутствии `ИННФЛ` и `ФИО` — элемент `ИныеСвед` **обязателен**.
- [ ] При отсутствии `ФИО` — `ИныеСвед` **обязателен**.
- [ ] При отсутствии `ИныеСвед` и `ИННФЛ` — `ФИО` **обязателен**.

### 15.11 `СвИнНеУч` — условная обязательность (Schematron)

- [ ] Хотя бы один из `Идентиф` или `ИныеСвед` **обязателен** (Schematron: код 0400300003).
- [ ] Оба отсутствуют — **ошибка**.
- [ ] `ИдСтат`: допустимые значения `ИО` или `ИГ` — иное отклоняется.
- [ ] `КодСтр` (ОКСМТип): ровно 3 цифры.

### 15.12 `ИнфПол` — информационные поля

- [ ] `ИдФайлИнфПол`: ровно 36 символов — 35 и 37 символов отклоняются.
- [ ] `ТекстИнф/@Идентиф`: минимум 1, максимум 50 символов.
- [ ] `ТекстИнф/@Значение`: минимум 1, максимум 2000 символов.
- [ ] Элемент `ИнфПол` необязателен — отсутствие принимается.
